## Overview

RAP ships as a single Docker image. The Dockerfile uses a two-stage build:

1. **Stage 1 — `frontend-builder` (Node 22 Alpine):** Installs dependencies and runs `npm run build`, producing a compiled Vite/React bundle in `frontend/dist/`.
2. **Stage 2 — Python 3.13 slim:** Installs Python dependencies, copies the backend source and the compiled frontend assets, and serves everything through uvicorn.

The result is a self-contained image that serves both the API and the React SPA from a single port.

## Docker

### Build the image

Run this from the repository root (where the `Dockerfile` lives):

```bash
docker build -t rap-app .
```

### Run the container

```bash
docker run -p 8000:8000 \
  -e OPENAI_API_KEY=sk-... \
  -e DATABASE_URL=postgresql://user:password@host/dbname?sslmode=require \
  rap-app
```

The application starts on port 8000. Open `http://localhost:8000` to reach the React app and `http://localhost:8000/docs` for the API reference.

### Override the port

The default port is `8000`, set via `ENV PORT=8000` in the Dockerfile. To run on a different port, override the `PORT` environment variable:

```bash
docker run -p 3000:3000 \
  -e PORT=3000 \
  -e OPENAI_API_KEY=sk-... \
  -e DATABASE_URL=postgresql://... \
  rap-app
```

The `CMD` in the Dockerfile uses shell form so `${PORT}` is expanded correctly at runtime:

```dockerfile
CMD uvicorn main:app --host 0.0.0.0 --port ${PORT}
```

### Health check

The image includes a built-in Docker health check that polls the `/api/health` endpoint:

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD sh -c 'curl -fsS "http://127.0.0.1:${PORT}/api/health" || exit 1'
```

You can also check health manually:

```bash
curl http://localhost:8000/api/health
```

Expected response from a healthy container:

```json
{
  "status": "healthy",
  "services": {
    "agent": "healthy",
    "api_key": "configured",
    "data_directory": "accessible",
    "chroma_database": "accessible"
  },
  "metrics": {
    "active_sessions": 0,
    "legacy_sessions": 0
  }
}
```

### Security: non-root user

The image creates and switches to `appuser` (UID 1000) before the application starts. Do not override this with `--user root` in production.

```dockerfile
RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
USER appuser
```

## Environment variables

Set these when running the container. None have defaults suitable for production except `PORT`.

| Variable | Required | Default | Description |
|---|---|---|---|
| `OPENAI_API_KEY` | Yes | — | OpenAI API key for the LLM agent and embeddings |
| `DATABASE_URL` | Yes | hardcoded Neon URL in `config.py` | PostgreSQL connection string |
| `PORT` | No | `8000` | Port uvicorn listens on |
| `PRIMARY_LLM_MODEL` | No | `gpt-5-mini-2025-08-07` | Model used for conversation turns |
| `REPORT_LLM_MODEL` | No | `gpt-5-2025-08-07` | Model used for report generation |
| `EMBEDDING_MODEL` | No | `text-embedding-3-small` | OpenAI embedding model for ChromaDB |
| `MAX_QUESTIONS_PER_COMPETENCY` | No | `5` | Maximum questions asked per competency |
| `PASSING_SCORE_THRESHOLD` | No | `60.0` | Minimum percentage score to pass a competency |
| `ASSESSMENT_TIMEOUT_MINUTES` | No | `120` | Maximum session duration in minutes |
| `SESSION_TTL_MINUTES` | No | `60` | Inactivity timeout before a session is cleaned up |
| `GRAPH_PROCESSING_TIMEOUT` | No | `150` | LangGraph processing timeout in seconds |

<Warning>
  The default `DATABASE_URL` in `config.py` points to a shared Neon development database. Always set your own `DATABASE_URL` in production to avoid data collisions.
</Warning>

## Railway

Railway is the recommended hosting platform for RAP. It detects the `Dockerfile` automatically and sets `PORT` as a managed environment variable.

<Steps>
  <Step title="Connect your repository">
    In the Railway dashboard, create a new project and select **Deploy from GitHub repo**. Choose the `RAP_RS` repository.

    Railway detects the `Dockerfile` in the repository root and uses it to build the image.
  </Step>

  <Step title="Add environment variables">
    In the Railway service settings, go to **Variables** and add:

    ```env
    OPENAI_API_KEY=sk-...
    DATABASE_URL=postgresql://user:password@host/dbname?sslmode=require
    ```

    Do not set `PORT` — Railway injects it automatically.
  </Step>

  <Step title="Deploy">
    Push to your connected branch. Railway builds the image and deploys it. The service URL is shown in the dashboard once the health check passes.
  </Step>

  <Step title="Verify the deployment">
    Open `https://your-service.up.railway.app/api/health`. A `"status": "healthy"` response confirms the backend is running correctly.

    Open `https://your-service.up.railway.app` to reach the React frontend.
  </Step>
</Steps>

<Note>
  Railway sets `PORT` dynamically per deployment. The Dockerfile's shell-form `CMD` and health check both reference `${PORT}` so they work correctly without any manual configuration.
</Note>

## Docker Compose (local development)

For local development with a local PostgreSQL instance, you can use a minimal `docker-compose.yml`:

```yaml
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: rap
      POSTGRES_PASSWORD: rap
      POSTGRES_DB: rapdb
    ports:
      - "5432:5432"

  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      OPENAI_API_KEY: ${OPENAI_API_KEY}
      DATABASE_URL: postgresql://rap:rap@db/rapdb
    depends_on:
      - db
```

```bash
OPENAI_API_KEY=sk-... docker compose up
```

<Note>
  This compose file is not included in the repository. Create it in the repository root if you need it for local container testing.
</Note>
