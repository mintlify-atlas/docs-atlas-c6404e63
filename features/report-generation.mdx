title: "Report Generation"
description: "How RAP generates, caches, and delivers personalised student reports after an assessment is completed."
---

Once an applicant finishes the 8-question assessment, the **`ReportGenerator`** uses an LLM to produce a `StudentReport` that summarises strengths, areas for improvement, and a personalised learning path.

## How a Report Is Generated

`ReportGenerator.generate_student_report()` in `src/tools/report_generator.py` works as follows:

1. Groups all `QuestionResponse` records by `competency_id` and calculates per-competency average scores.
2. Builds a concise prompt that includes the applicant's professional profile and the per-competency summary.
3. Sends the prompt to the report LLM model (`gpt-5-2025-08-07` by default, up to 4 096 tokens) with a **100-second timeout**.
4. Parses the JSON response and wraps it in a `StudentReport` object.
5. The report is cached at `ConversationState.misc["generated_report"]` so it is never regenerated for the same session.

<Warning>
  If the LLM response cannot be parsed as valid JSON, `generate_student_report` raises an
  exception — there is no silent fallback. Ensure the model is reachable before starting an
  assessment session.
</Warning>

## The `StudentReport` Model

Defined in `src/schemas/models.py`:

```python
class StudentReport(BaseModel):
    assessment_id: str
    applicant_name: str
    program_name: str
    overall_score: float                              # Average of all LLM scores (1–5 scale)
    strengths: List[str]
    areas_to_improve: List[str]
    personalized_path: List[PersonalizedRecommendation]
    estimated_completion_time: str
    next_steps: List[str]
    generated_at: datetime
```

Each item in `personalized_path` is a `PersonalizedRecommendation`:

```python
class PersonalizedRecommendation(BaseModel):
    competency_id: str
    action: str        # e.g. "skip", "take_course", "take_micro_credential", "review"
    reason: str
    suggested_courses: List[str]
    estimated_time_hours: Optional[int]
```

## Retrieving a Report

After the chat step signals `report_ready=True`, the frontend calls:

```
GET /api/report/{session_id}
```

The endpoint (defined in `src/api.py`) checks `session_state.misc["generated_report"]` first. If the cached report exists, it is returned immediately. If not, `ReportGenerator.generate_student_report()` is called on-demand.

```
404  — session not found
400  — assessment not yet completed
200  — StudentReport JSON
```

## Report Caching

The report is stored in the in-memory session state immediately after generation:

```python
# vap_agent.py — _report_generation_node
state.misc["generated_report"] = student_report
```

And served from cache by the API:

```python
# api.py — get_report endpoint
if "generated_report" in session_state.misc:
    return session_state.misc["generated_report"]
```

<Note>
  Because sessions are held in memory, a server restart will clear all cached reports. In
  production, consider persisting reports to the PostgreSQL database alongside the
  `FinalEvaluation` records.
</Note>
